## 1. Stack & Architecture

- **Runtime**: Node.js + NestJS
- **Language**: TypeScript (strict mode)
- **ORM**: Prisma with PostgreSQL
- **Patterns**: CQRS + Domain Events + Clean Architecture
- **Validation**: class-validator + class-transformer
- **Docs**: Swagger/OpenAPI via @nestjs/swagger
- **Testing**: Jest

### Architecture Layers

```
src/
├── domain/          # Pure business logic, no framework deps
│   ├── entities/    # Domain entities (Client, Product, Invoice, etc.)
│   ├── enums/       # Typed enums (InvoiceStatus, TaxType, etc.)
│   ├── value-objects/# Immutable value objects
│   ├── exceptions/  # Domain-specific exceptions
│   └── repositories/ # Repository interfaces (abstractions)
│
├── application/     # CQRS orchestration
│   ├── {aggregate}/ # e.g., clients, products, invoices
│   │   ├── commands/ # Write operations (Create, Update, Delete)
│   │   ├── queries/  # Read operations (Get, List)
│   │   └── handlers/ # CommandHandler / QueryHandler implementations
│   └── {feature}/   # Cross-cutting commands/queries
│       ├── commands/
│       └── handlers/
│
├── infrastructure/  # External dependencies
│   ├── common/      # Filters, interceptors, decorators
│   │   ├── filters/
│   │   └── interceptors/
│   └── persistence/  # Prisma implementation
│       └── prisma/
│           ├── mappers/    # Domain ↔ Prisma entity mappers
│           └── repositories/ # Repository implementations
│
└── presentation/    # API layer
    ├── {module}/     # e.g., clients, products, invoices
    │   ├── dto/      # Request/Response DTOs
    │   └── controllers/
    └── controllers/  # Root controllers
```

## 2. Naming Conventions

### Files

| What | Convention | Example |
|---|---|---|
| Entity | `PascalCase` singular | `Client.ts` |
| Value Object | `PascalCase` | `Money.ts` |
| Enum | `PascalCase` singular | `InvoiceStatus.ts` |
| Repository impl | `PascalCase` + suffix | `PrismaClientRepository.ts` |
| Mapper | `PascalCase` + suffix | `ClientMapper.ts` |
| Command | `PascalCase` + suffix | `CreateClientCommand.ts` |
| Query | `PascalCase` + suffix | `GetClientQuery.ts` |
| Handler | `PascalCase` + suffix | `CreateClientHandler.ts` |
| DTO | `PascalCase` + suffix | `CreateClientDto.ts` |
| Controller | `PascalCase` + suffix | `ClientsController.ts` |
| Service | `PascalCase` + suffix | `ClientsService.ts` |
| Exception | `PascalCase` + suffix | `ClientNotFoundException.ts` |
| Test | Same as target + `.spec.ts` | `create-client.handler.spec.ts` |

### Code Identifiers

| What | Convention | Example |
|---|---|---|
| Class / Interface / Type / Enum | `PascalCase` | `class ClientRepository` |
| Method / Function / Variable | `camelCase` | `createClient()` |

| Private field | `_camelCase` | `private _logger` |
| Parameter | `camelCase` | `(id: string)` |
| Generic type | `PascalCase` single letter or `TEntity` | `TResult`, `TEntity` |
| Enum member | `UPPER_SNAKE_CASE` | `InvoiceStatus.PAID` |
| Boolean variable | `is`/`has`/`should` prefix | `isActive`, `hasPendingInvoice` |

### Database (Prisma)

| What | Convention | Example |
|---|---|---|
| Model | `PascalCase` singular | `Client` |
| Field | `camelCase` | `createdAt` |
| Relation | `camelCase` singular | `client`, `items` |
| Enum value | `UPPER_SNAKE_CASE` | `PAID` |

## 3. Git Workflow

### Branch Naming

```
{type}/{ticket-id}-{short-description}
```

| Type | Use for | Example |
|---|---|---|
| `feat/` | New features | `feat/POS-42-user-authentication` |
| `fix/` | Bug fixes | `fix/POS-87-invoice-tax-calculation` |
| `refactor/` | Code restructuring | `refactor/POS-23-move-to-cqrs` |
| `docs/` | Documentation only | `docs/POS-91-api-contract` |
| `chore/` | Tooling, deps, config | `chore/POS-95-upgrade-prisma` |
| `test/` | Tests only | `test/POS-88-client-handler` |

### Commit Messages

Follow [Conventional Commits](https://www.conventionalcommits.org/):

```
{type}({scope}): {description}

[optional body]

[optional footer]
```

| Type | When to use |
|---|---|
| `feat` | New feature |
| `fix` | Bug fix |
| `refactor` | Code change that neither fixes a bug nor adds a feature |
| `test` | Adding or fixing tests |
| `docs` | Documentation changes |
| `chore` | Build process, tooling, dependency updates |
| `perf` | Performance improvements |
| `ci` | CI/CD changes |

**Examples:**
```
feat(clients): add client creation with tax identification
fix(invoices): correct tax calculation for exempt items
refactor(products)!: migrate inventory to warehouse-aggregate
docs(api): document invoice endpoints and status codes
```

### Pull Request Titles

Same format as commits. Use squash-merge to keep history linear.

### Issues

```
[POS-{id}] {short description}
[POS-42] Client creation endpoint returns wrong status code
```

## 4. API REST Contracts

### URL Structure

```
/api/v{version}/{resource}/{action}
```

| Element | Rule |
|---|---|
| Base | `/api/v1` |
| Resource | Plural, kebab-case | `clients`, `products` |
| Action | Prefer HTTP verb, not path segment | `POST /clients`, not `POST /clients/create` |
| IDs | Path parameter | `GET /clients/:id` |
| Query params | Filtering, pagination | `GET /clients?status=active&page=1` |

### HTTP Methods

| Method | Use | Returns |
|---|---|---|
| `GET` | Retrieve single or list | `200` with body |
| `POST` | Create resource | `201` with body + `Location` header |
| `PATCH` | Partial update | `200` with body |
| `PUT` | Full replace | `200` with body |
| `DELETE` | Remove resource | `204` no body |

### Status Codes

| Code | When |
|---|---|
| `200` | Successful read/update |
| `201` | Resource created |
| `204` | Successful delete (no body) |
| `400` | Validation error, malformed request |
| `401` | Not authenticated |
| `403` | Not authorized |
| `404` | Resource not found |
| `409` | Conflict (e.g., duplicate key) |
| `422` | Business rule violation |
| `500` | Internal server error |

### Error Response Format

```json
{
  "statusCode": 400,
  "message": "Validation failed",
  "errors": [
    {
      "field": "taxId",
      "message": "Tax ID must be 11 digits"
    }
  ],
  "timestamp": "2026-05-19T20:30:00.000Z",
  "path": "/api/v1/clients"
}
```

### Versioning

- Version in URL path: `/api/v1/`
- Breaking changes → bump version
- Non-breaking additions → same version
- Never remove existing fields from responses (add new ones)

## 5. Code Rules

### Imports Order (enforced by ESLint or manual)

```typescript
// 1. Node built-ins
// 2. External packages
// 3. Internal packages (nestjs, prisma)
// 4. Domain layer
// 5. Application layer
// 6. Infrastructure layer
// 7. Presentation layer
// 8. Relative imports
```

### Class Structure

```typescript
// 1. Decorators (@Entity, @Injectable, etc.)
// 2. Class name
// 3. Implements clause
// 4. Class decorators (@Prop)
// 5. Static readonly fields
// 6. Instance readonly fields
// 7. Constructor parameters (with accessors)
// 8. Public methods
// 9. Protected methods
// 10. Private methods
// 11. Public getters/setters
```

### CQRS Rules

- **Commands** write state → return `void` or `Result<T>`
- **Queries** read state → return `Promise<T>` directly
- **Handlers** inject repository via constructor
- **Commands** do NOT call queries (no read in write path)
- **Query handlers** do NOT mutate state
- Domain events dispatched from command handlers via `EventPublisher`

### DTO Rules

- Request DTOs: `CreateXxxDto`, `UpdateXxxDto`, `XxxQueryDto`
- Response DTOs: `XxxResponseDto`, `XxxListResponseDto`
- Use `class-validator` decorators for validation
- Use `@nestjs/swagger` decorators for OpenAPI docs
- NO business logic in DTOs — only validation and transformation

### General

- **Prefer `const`** over `let`; never use `var`
- **Prefer `async/await`** over raw `.then()` chains
- **Never use `any`** without explicit justification comment
- **Prefer `interface`** over `type` for object shapes
- **Use `unknown`** for external data; narrow before use
- **Prefer named exports** over default exports for better refactoring
- **Use absolute imports** via `tsconfig.json` paths (`@domain/`, `@application/`, `@infrastructure/`, `@presentation/`)
- **Prefer `Result<T>` pattern** for operations that can fail, over throwing exceptions for expected errors
- **Throw exceptions** only for truly unexpected errors (system failures)

## 7. Testing Rules

- **Unit tests** for all handlers, services, and domain logic
- **Coverage minimum**: 80% (enforced in CI)
- **Naming**: `{method}.{scenario}.{expected behavior}`
- **Arrange-Act-Assert** pattern in test structure
- **Mock** external dependencies (repositories, external services)
- **Do not mock domain objects** — test real domain logic
